一、整体架构与核心思路
1.1 核心设计理念
建议采用“扫描引擎 + 数据模型 + 暴露端点”的三层架构：

层次	职责
扫描引擎层	在应用启动时，通过Spring的ApplicationListener或@PostConstruct触发扫描，收集所有接口的元数据
数据模型层	定义统一的接口信息模型（ApiInfo），包含接口类型、路径、方法、参数、返回值等
暴露端点层	通过自定义的HTTP端点（如/actuator/apis）将收集到的信息以JSON格式返回
1.2 技术选型建议
构建工具：同时支持Maven和Gradle，发布到Maven中央仓库

Spring Boot版本：支持2.x和3.x（注意Jakarta EE的包路径变化）

核心依赖：spring-boot-starter-web、spring-boot-starter-actuator、spring-boot-starter-websocket

二、HTTP RESTful接口扫描
2.1 扫描方式
方案一：运行时Bean扫描（推荐）

通过Spring的ApplicationContext获取所有标注了@RestController或@Controller的Bean，再通过反射获取每个方法上的@RequestMapping、@GetMapping等注解信息：

java
@Autowired
private ApplicationContext applicationContext;

public List<ApiInfo> scanRestControllers() {
Map<String, Object> controllers = applicationContext.getBeansWithAnnotation(RestController.class);
// 遍历每个Controller，解析方法上的Mapping注解
}
方案二：源码静态扫描

参考SpringBoot-API-Scanner的思路，使用JavaParser解析源码的抽象语法树（AST），精准提取注解信息。这种方式不依赖运行时容器，但实现复杂度较高。

2.2 可提取的信息
类级别路径（@RequestMapping的value）

方法级别路径、HTTP方法（GET/POST/PUT/DELETE等）

参数信息（@RequestParam、@PathVariable、@RequestBody等）

返回值类型

方法上的其他注解（如@PreAuthorize权限注解）

三、WebSocket接口扫描
3.1 扫描方式
Spring Boot中WebSocket有两种主流实现方式：

方式一：@ServerEndpoint注解方式

通过ServerEndpointExporter在启动时扫描所有带@ServerEndpoint注解的类。可以自定义一个扫描器，遍历所有标注了该注解的Bean：

java
Map<String, Object> endpoints = applicationContext.getBeansWithAnnotation(ServerEndpoint.class);
// 解析每个端点的path、消息处理方法等
方式二：WebSocketHandler注册方式

通过WebSocketConfigurer或WebSocketMessageBrokerConfigurer手动注册端点。可以从WebSocketHandlerRegistry中获取已注册的端点信息。

3.2 可提取的信息
WebSocket端点路径（如/ws/socket）

支持的消息类型（Text/Binary）

订阅的Topic/Queue（STOMP协议下）

四、RPC接口扫描
4.1 Dubbo接口
Dubbo提供了多种获取暴露服务信息的方式：

方式一：通过ServiceConfig API

```java
ServiceConfig serviceConfig = new ServiceConfig();
List<ServiceConfig> serviceConfigs = serviceConfig.getServiceConfigs();
for (ServiceConfig config : serviceConfigs) {
String interfaceName = config.getInterfaceName();
List<MethodConfig> methods = config.getMethodConfigs();
// 获取每个方法的入参、出参信息
}
```

方式二：通过MetadataService

Dubbo的MetadataService提供了getExportedURLs等方法，可以获取已暴露服务的元数据信息。

方式三：通过Dubbo SPI扩展

实现ServiceBean的onApplicationEvent方法，监听服务暴露事件。

4.2 gRPC接口
gRPC提供了Server Reflection机制，可以在运行时查询服务定义：

```java
// 启用反射服务
Server server = ServerBuilder.forPort(port)
.addService(new YourServiceImpl())
.addService(new ProtoReflectionService())  // 关键：注册反射服务
.build();
```

通过反射客户端可以获取所有服务的ServiceDescriptor，包含服务名、方法名、输入输出消息类型等信息。

4.3 可提取的信息
服务名称（接口全限定名）

方法列表及方法签名（入参类型、出参类型）

协议类型（Dubbo协议、Triple协议等）

版本号、分组信息

六、关键技术与注意事项

Spring Boot 2.x vs 3.x：注意javax vs jakarta包路径的差异

响应式应用：支持spring-webflux的@RestController

多协议共存：一个项目可能同时有REST、WebSocket和RPC，需要统一扫描

6.3 性能与安全
扫描应在应用启动时完成，避免运行时反复扫描

扫描结果可缓存，避免重复计算

暴露的接口信息端点应添加权限控制，避免敏感信息泄露

七、总结
实现这样一个依赖库的核心技术路径是：

REST接口：通过Spring的ApplicationContext获取所有@RestController Bean，反射解析Mapping注解

WebSocket接口：扫描@ServerEndpoint注解或从WebSocketHandlerRegistry获取注册信息

RPC接口：Dubbo通过ServiceConfig API或MetadataService获取；gRPC通过Server Reflection机制

统一暴露：通过自定义Actuator端点或普通Controller将数据以JSON格式返回

前端展示：参考Swagger UI的设计，按接口类型分类展示，支持搜索和在线调试

这个方案的核心优势在于运行时动态扫描，无需用户额外配置，真正做到“引入即用”。

二、TR接口接入方案
2.1 什么是TR接口？
TR（Trace RPC）通常指基于Dubbo/ gRPC/ tRPC等协议的RPC接口。这里我理解为需要支持腾讯tRPC或通用RPC框架的链路追踪。

2.2 TR接口扫描实现
```java
@Component
public class TrInterfaceScanner {

    @Autowired(required = false)
    private ApplicationContext applicationContext;
    
    public List<TrApiInfo> scanTrInterfaces() {
        List<TrApiInfo> result = new ArrayList<>();
        
        // 1. 扫描 tRPC 服务（如果项目中引入了 tRPC）
        try {
            scanTrpcServices(result);
        } catch (Exception e) {
            log.debug("tRPC not detected, skip scanning");
        }
        
        // 2. 扫描 Dubbo 服务
        try {
            scanDubboServices(result);
        } catch (Exception e) {
            log.debug("Dubbo not detected, skip scanning");
        }
        
        // 3. 扫描 gRPC 服务
        try {
            scanGrpcServices(result);
        } catch (Exception e) {
            log.debug("gRPC not detected, skip scanning");
        }
        
        return result;
    }
    
    private void scanTrpcServices(List<TrApiInfo> result) {
        // 通过 tRPC 的 @TRpcService 注解扫描
        Map<String, Object> beans = applicationContext.getBeansWithAnnotation(TRpcService.class);
        // 解析服务名、方法列表、参数类型等信息
    }
}
```

3.2 TraceId生成与传递
3.2.1 TraceId生成器（可配置策略）

}

}
}

// 用户可自定义实现，例如使用Snowflake算法

3.2.2 过滤器实现
```java
@Component
public class TraceIdFilter implements Filter {

    @Autowired
    private TraceIdGenerator traceIdGenerator;
    
    @Autowired(required = false)
    private TraceIdResolver traceIdResolver;  // 用户自定义解析器
    
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, 
                         FilterChain chain) {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        
        // 1. 尝试从请求头获取traceId（用户自定义）
        String traceId = httpRequest.getHeader("X-Trace-Id");
        
        // 2. 如果用户配置了自定义解析器
        if (traceIdResolver != null) {
            traceId = traceIdResolver.resolve(request);
        }
        
        // 3. 如果都没有，则自动生成
        if (StringUtils.isEmpty(traceId)) {
            traceId = traceIdGenerator.generate();
        }
        
        // 4. 放入MDC
        MDC.put("traceId", traceId);
        MDC.put("spanId", generateSpanId());
        
        // 5. 放入响应头，方便前端获取
        httpResponse.setHeader("X-Trace-Id", traceId);
        
        try {
            chain.doFilter(request, response);
        } finally {
            // 清除MDC，避免内存泄漏
            MDC.clear();
        }
    }
}
```

3.2.3 跨线程传递（重点）
使用装饰器模式包装线程池，实现MDC上下文传递：

java
```@Component
public class MdcThreadPoolTaskExecutor extends ThreadPoolTaskExecutor {

    @Override
    public void execute(Runnable task) {
        // 获取当前线程的MDC上下文
        Map<String, String> context = MDC.getCopyOfContextMap();
        super.execute(() -> {
            try {
                // 将上下文设置到子线程
                if (context != null) {
                    MDC.setContextMap(context);
                }
                task.run();
            } finally {
                MDC.clear();
            }
        });
    }
}
```

或者使用@Async异步任务传递：

```java
@Configuration
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new MdcThreadPoolTaskExecutor();
        // ... 配置参数
        return executor;
    }
}
```

对于CompletableFuture等异步编程：

```java
// 使用Supplier包装
public static <T> Supplier<T> wrapWithMdc(Supplier<T> supplier) {
    Map<String, String> context = MDC.getCopyOfContextMap();
        return () -> {
            try {
                if (context != null) {
                    MDC.setContextMap(context);
                }
                return supplier.get();
            } finally {
            MDC.clear();
        }
    };
}
```

```java

    log.info("异步任务执行");
    return "result";
}), executor);
```

四、日志配置管理
4.1 默认logback配置（作为fallback）
在依赖的src/main/resources目录下放置logback-spring.xml：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <!-- 定义日志格式，包含traceId -->
    <property name="LOG_PATTERN" 
              value="%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] [%X{traceId:-}] %-5level %logger{36} - %msg%n"/>

    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>${LOG_PATTERN}</pattern>
        </encoder>
    </appender>
    
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>${LOG_PATH:-./logs}/application.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>${LOG_PATH:-./logs}/application.%d{yyyy-MM-dd}.log</fileNamePattern>
            <maxHistory>30</maxHistory>
        </rollingPolicy>
        <encoder>
            <pattern>${LOG_PATTERN}</pattern>
        </encoder>
    </appender>
    
    <!-- 默认日志级别 -->
    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="FILE"/>
    </root>
</configuration>
```

4.2 智能加载策略
核心类：自定义日志配置加载器

```java
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class LogConfigLoader implements ApplicationListener<ApplicationEnvironmentPreparedEvent> {

    @Override
    public void onApplicationEvent(ApplicationEnvironmentPreparedEvent event) {
        // 检测用户是否配置了logback-spring.xml
        boolean userHasLogConfig = detectUserLogConfig();
        
        if (!userHasLogConfig) {
            // 使用Contra的默认配置
            System.setProperty("logback.configurationFile", 
                "classpath:contra-logback-spring.xml");
            log.info("Contra: Using default logback configuration");
        } else {
            log.info("Contra: User provided logback configuration, using it");
        }
    }
    
    private boolean detectUserLogConfig() {
        // 检查classpath下是否存在logback-spring.xml
        ClassLoader classLoader = Thread.currentThread().getContextClassLoader();
        return classLoader.getResource("logback-spring.xml") != null ||
               classLoader.getResource("logback.xml") != null;
    }
}
```

7.2 用户使用示例
1. 引入依赖：

```xml
<dependency>
    <groupId>io.github.yourname</groupId>
    <artifactId>contra-spring-boot-starter</artifactId>
    <version>1.0.0</version>
</dependency>
```

2. 配置（可选）：

```yaml
contra:
    trace:
        enabled: true
        header-name: X-Trace-Id
    log:
      pattern: "%d [%X{traceId}] %msg%n"
```

3. 自定义TraceId生成：

```java
@Component
public class MyTraceIdGenerator implements TraceIdGenerator {
    @Override
    public String generate() {
        // 使用Snowflake算法
        return snowflake.nextId();
    }
}
```
