## 一、项目概述

**Archimedes** 是一个"引入即用"的 Spring Boot API 可观测性依赖（Maven 坐标 `io.github.nianliu:archimedes-*`）。宿主应用只需引入一个 starter、零配置启动，即可获得两大能力：

1. **接口契约自动扫描与展示**：自动发现宿主应用的 REST、WebSocket、Dubbo、gRPC、SOFARPC-TR、腾讯 tRPC 全部接口契约，提供 JSON 端点与内置可视化 UI（协议分 Tab、搜索筛选、REST 在线调试）。
2. **全链路日志追踪**：为每个 HTTP 请求自动建立 traceId 上下文（可接入宿主自有 traceId、可替换生成算法），跨线程自动传递，并支持按 traceId 一键查询该请求链路上所有线程产生的全部日志；宿主没有 logback 配置时提供带 traceId 的内置日志配置兜底。

核心设计原则贯穿全部实现：

| 原则 | 落地方式 |
|---|---|
| 引入即用 | 全部能力自动装配，无需任何注解/配置/代码侵入；每项能力均有独立开关可关闭 |
| 零影响 | 所有协议依赖 `<optional>` + `@ConditionalOnClass` 条件装配，宿主没用到的框架完全不装配对应 Bean |
| 可替换 | 关键实现均为 SPI（`@ConditionalOnMissingBean` 让位）：traceId 生成/解析、日志存储等宿主注册同类型 Bean 即替换 |
| 不破坏宿主 | MDC 按 key 精准恢复绝不 `MDC.clear()`；日志兜底在宿主有任何自有配置时完全不介入；包装线程池保持 Bean 类型不变 |

## 二、总体架构

### 2.1 模块结构（Maven 多模块）

| 模块 | 职责 | 关键约束 |
|---|---|---|
| `archimedes-parent` | 聚合与版本属性管理 | **不导入任何 Spring Boot BOM**（两套 BOM 必须按模块隔离，防版本污染） |
| `archimedes-core` | 框架无关核心：数据模型、全部扫描器、端点控制器、trace/日志核心、UI 资源 | 按 SB 2.7 基线编译、**Java 8 字节码**、禁止 servlet API 依赖 → **单 jar 被两个 starter 复用** |
| `archimedes-spring-boot-2-starter` | SB 2.7.x（javax）薄注册层 | 经典 `@Configuration + @AutoConfigureAfter`，`META-INF/spring.factories` 注册 |
| `archimedes-spring-boot-3-starter` | SB 3.x（jakarta）薄注册层 | `@AutoConfiguration`，`META-INF/spring/...AutoConfiguration.imports` 注册 |
| `example` / `example-boot2` | SB3 / SB2.7 示例应用（端口 8080/8081） | 验证"引入即用"真机效果 |
| `example-all` | **全功能测试项目**（端口 8082） | 一个应用覆盖全部协议与追踪能力，附全量端到端测试（见第五章） |

### 2.2 双 Spring Boot 版本支持机制

- **单 core jar 双端复用**：core 只依赖两大版本间 API 一致的 Spring 类（`HandlerMethod`、`RequestMappingHandlerMapping` 形状、logback `AppenderBase` 等），凡涉及 javax/jakarta 分叉的代码（Servlet Filter 壳、`@ServerEndpoint` 扫描）下沉到各 starter 各写一份薄壳，公共逻辑仍在 core。
- **双注册机制**：SB2 走 `spring.factories`，SB3 走 `AutoConfiguration.imports`；core 自身的 `EnvironmentPostProcessor`（日志兜底）在 `spring.factories` 注册一份即可双端通用。
- **版本矩阵**：SB2 starter 最低 Java 8；SB3 starter 最低 Java 17；Web 栈支持 Servlet 与 Reactive（WebFlux）双栈。

### 2.3 自动装配拓扑

```
ArchimedesAutoConfiguration          (SERVLET 门控)  → RestApiScanner + ApiController + WS 扫描×3 + RPC 扫描×4(共享@Import)
ArchimedesReactiveAutoConfiguration  (REACTIVE 门控) → ReactiveRestApiScanner + ApiController + RPC 扫描×4(共享@Import)
ArchimedesTraceAutoConfiguration     (SERVLET 门控)  → TraceIdFilter + TraceContextManager + MDC 跨线程 BPP + 覆盖边界检测
ArchimedesLogAutoConfiguration       (SERVLET 门控)  → 采集 Appender 挂载 + LogStore + 日志查询端点
ArchimedesLoggingEnvironmentPostProcessor (栈无关，早于日志系统初始化) → logback 兜底注入
```

SERVLET 与 REACTIVE 分支条件互斥；四类 RPC 扫描配置与 Web 栈无关，抽为共享配置被两分支 `@Import` 复用。

## 三、功能清单与实现方式

### 3.1 接口契约扫描与展示

#### REST（Servlet + WebFlux 双栈）

| 功能 | 实现方式 |
|---|---|
| 扫描 `@RestController` 全量契约：路径（类/方法级合并）、HTTP 方法、参数（`@RequestParam`/`@PathVariable`/`@RequestBody`/`@RequestHeader`，含来源与 required）、返回类型（含泛型签名）、consumes/produces、`@Deprecated` 标记 | 容器内 `RequestMappingHandlerMapping.getHandlerMethods()` 自省——**读 Spring 已解析好的路由表而非自行反射注解**，天然与宿主实际生效路由一致；Servlet 侧兼容 PathPattern（5.3+ 默认）与 AntPathMatcher 两种路径条件 |
| WebFlux 响应式端点扫描（`Mono`/`Flux` 返回类型签名） | 抽象骨架 `AbstractRestApiScanner`（缓存/参数提取/排除规则栈无关）+ `ReactiveRestApiScanner` 消费 reactive `RequestMappingHandlerMapping`；REACTIVE 条件装配分支独立注册 |
| 排除规则 | 自身端点（`{base-path}/**`）、框架 `/error` 不出现在结果；`base-packages` 配置可按包前缀过滤 |
| 结果缓存 | 首次请求扫描后缓存不可变列表（路由表启动后不变） |

#### WebSocket（三种主流形态，Servlet 栈）

| 形态 | 实现方式 |
|---|---|
| `@ServerEndpoint` 注解端点 | javax/jakarta 包名分叉 → 扫描器各落 SB2/SB3 starter（须为 Spring Bean） |
| `WebSocketConfigurer` 注册的 handler | 自省 `SimpleUrlHandlerMapping` 中的 WebSocket handler 映射，识别 SockJS 标记 |
| STOMP | 自省 `SimpAnnotationMethodMessageHandler` 提取 `@MessageMapping`/`@SubscribeMapping` 目的地 + 握手端点 |

统一通过 **`WebSocketApiContributor` SPI** 汇入控制器聚合（该 SPI 模式也是后续 RPC 扫描器的模板）。

#### RPC 四协议

| 协议 | 实现方式 | 关键决策 |
|---|---|---|
| Dubbo | 容器内 `ServiceBean` 自省，提取接口 FQCN、version、group、方法签名 | 同时覆盖 `@DubboService` 注解与 XML 注册两种形态；Dubbo 2.7/3.x 兼容 |
| gRPC | `BindableService.bindService()` 返回的 `ServerServiceDefinition` 自省，提取服务名、方法、streaming 形态、消息类型 | **不依赖 Server Reflection**（宿主无需注册反射服务），只需 `grpc-api` 轻依赖 |
| SOFARPC-TR | 反射读取 `@SofaService` 注解 Bean（interfaceType/uniqueId/bindings） | **零 SOFA 编译依赖**：`@ConditionalOnClass(name="...")` 字符串条件 + `AnnotatedElementUtils` 按 FQCN 反射读取 |
| 腾讯 tRPC | 反射读取 `@TRpcService`（name/version/group） | 同上零依赖；注解 FQCN 按开源 tRPC-Java 假设，不匹配时安全不装配 |

四协议共用统一模型 `RpcApiInfo`（`protocol` 字段区分 DUBBO/GRPC/SOFA_TR/TRPC，`metadata` 承载协议特有信息），通过 **`RpcApiContributor` SPI** 聚合。

#### JSON 端点与内置 UI

- `GET {base-path}/apis`：分组 JSON `{restApis, webSocketApis, rpcApis}`，协议不存在时输出空数组而非缺失。
- `GET {base-path}`：内置 UI（单 HTML 文件、零构建、零外部 CDN，内网可用）：
  - **协议 Tab**：REST / WebSocket / RPC / TR / Trace Logs 五 Tab（TR 与 RPC 按 `protocol` 二分），标签带计数，hash 记忆当前页；
  - **搜索筛选**：全局文本过滤 + REST 方法下拉 + RPC/TR 协议 chip；
  - **REST 在线调试**：条目行内展开 try-it 面板，按契约预填路径变量/Query/Header/Body，浏览器同源 fetch 发起，展示状态码/耗时/响应头/响应体（JSON 美化）；响应头携带 trace 头时**一键跳转链路日志查询**——契约展示与链路追踪两大能力在 UI 内闭环。

### 3.2 全链路日志追踪

#### trace 上下文核心

| 功能 | 实现方式 |
|---|---|
| 每请求自动建立 traceId | Servlet Filter（`Ordered.HIGHEST_PRECEDENCE`，仅 REQUEST dispatch；javax/jakarta 双壳，逻辑在 core `TraceContextManager`） |
| traceId 解析链 | 用户 `TraceIdResolver` Bean → 请求头（`header-name`）→ 宿主 MDC（需 `use-project-trace-id=true`，实现"开启当前项目自身 traceId"需求）→ `TraceIdGenerator` 生成 |
| 可替换生成算法 | `TraceIdGenerator` SPI + 默认 UUID 实现（`@ConditionalOnMissingBean`，注册雪花算法等即替换） |
| spanId | 每请求生成并写入 MDC（key 可配置） |
| 响应头回写 | 默认把 traceId 写回 `X-Trace-Id` 响应头（前端/调用方可拿到） |
| 不破坏宿主 | `TraceScope` 记录写入前的旧值，请求结束按 key 精准恢复，**绝不 `MDC.clear()`** |

#### 跨线程 traceId 传递（覆盖 Spring 世界，无 javaagent）

| 场景 | 实现方式 |
|---|---|
| `@Async` / `ThreadPoolTaskExecutor` Bean | 自动注入 `MdcTaskDecorator`（反射注入私有 `taskDecorator` 字段，已有 decorator 自动组合），Bean 类型不变 |
| `Executor`/`ExecutorService`/`ScheduledExecutorService` 接口 Bean | `BeanPostProcessor` 自动替换为同接口 MDC 快照包装器；`TaskScheduler` 刻意跳过（定时任务不源于请求）；`exclude-beans` 逃生口 |
| `CompletableFuture` commonPool、自建裸线程池 | `MdcWrappers.wrap(...)` 一行式手动包装（六种函数形态） |
| 覆盖盲区预警 | 启动期检测 `@Async` 退化为非容器 `SimpleAsyncTaskExecutor` 的场景（宿主定义任意 Executor Bean 导致 Boot 默认池退避），打 WARN 并给出修复指引 |

**实现级决策**：采用纯 MDC 快照传递而未引入 TransmittableThreadLocal——无 javaagent 前提下 TTL 无增量收益，零新依赖达到同等覆盖（决策记录于归档 change）。

### 3.3 日志采集与按 traceId 查询

| 功能 | 实现方式 |
|---|---|
| 结构化采集 | 自定义 logback Appender **编程式挂载 root logger**（不依赖用户配置文件），只采集 MDC 含 traceId 的事件；**与输出格式完全解耦**——宿主改 pattern 不影响查询 |
| 存储 SPI | `LogStore` 接口（`append`/`queryByTraceId` 分页）；`@ConditionalOnMissingBean` 让位设计为 **Elasticsearch 等持久化实现预留接入位**（注册自定义 Bean 即接入） |
| 默认实现 | 内存有界存储：全局上限 10000 条 + 单链路 500 条（均可配），超限按最老链路整体淘汰；重启即失，定位为开发/排障场景 |
| 查询端点 | `GET {base-path}/logs/trace/{traceId}?page=&size=`（时间升序分页，含线程/级别/logger/spanId）；`GET {base-path}/trace/current`（当前请求 traceId） |
| UI | Trace Logs Tab：查询、时间线展示、**多线程按颜色区分**（直观看到一个请求横跨请求线程与异步线程的全部日志） |

### 3.4 日志配置管理（logback 兜底）

| 功能 | 实现方式 |
|---|---|
| 宿主无 logback 配置时自动兜底 | `EnvironmentPostProcessor`（顺序早于 Boot `LoggingApplicationListener`）注入 `logging.config` 指向内置 `archimedes-logback.xml`（控制台 + 滚动文件，pattern 含 `[traceId] [spanId]`） |
| 用户绝对优先 | 四条让位规则：`logging.config` 已设置 / 类路径存在任何约定名配置（`logback-spring.xml`、`logback.xml` 等）/ 日志系统非 logback / `fallback-enabled=false` —— 任一命中则完全不介入 |
| 防误拾取 | 内置文件刻意命名 `archimedes-logback.xml` 而非约定名，避免被 logback 自动发现机制错误加载 |
| 格式可配 | `archimedes.log.pattern/path/max-history/max-file-size` 经 `<springProperty>` 注入内置配置生效 |

## 四、使用指南

### 4.1 引入（按宿主 Spring Boot 大版本二选一）

```xml
<!-- Spring Boot 3.x（Java 17+） -->
<dependency>
    <groupId>io.github.nianliu</groupId>
    <artifactId>archimedes-spring-boot-3-starter</artifactId>
    <version>1.0-SNAPSHOT</version>
</dependency>

<!-- Spring Boot 2.7.x（Java 8+） -->
<dependency>
    <groupId>io.github.nianliu</groupId>
    <artifactId>archimedes-spring-boot-2-starter</artifactId>
    <version>1.0-SNAPSHOT</version>
</dependency>
```

引入后**无需任何配置**，启动即得：

| 端点 | 说明 |
|---|---|
| `GET /archimedes` | 内置 UI（Tab 浏览、筛选、REST 在线调试、链路日志查询） |
| `GET /archimedes/apis` | 接口契约分组 JSON |
| `GET /archimedes/logs/trace/{traceId}` | 按 traceId 查询链路全部日志（分页） |
| `GET /archimedes/trace/current` | 当前请求的 traceId |
| 每个 HTTP 响应 | 自动携带 `X-Trace-Id` 响应头 |

### 4.2 全量配置项

```yaml
archimedes:
  api:
    enabled: true            # 契约展示总开关
    base-path: /archimedes   # 端点根路径
    ui-enabled: true         # 是否挂载内置 UI
    base-packages: []        # 非空时只扫描这些包前缀下的 Controller
  trace:
    enabled: true            # 链路追踪总开关
    header-name: X-Trace-Id  # 透传/回写 traceId 的请求头名
    response-header: true    # 是否在响应头回写 traceId
    mdc-key: traceId         # traceId 的 MDC key
    span-id-key: spanId      # spanId 的 MDC key
    use-project-trace-id: false  # true=信任宿主自有 Filter 已写入的 MDC traceId
    propagation:
      enabled: true          # 跨线程自动传递开关
      exclude-beans: []      # 不希望被包装的 Executor Bean 名单
  log:
    capture:
      enabled: true              # 日志采集/查询开关
      max-entries: 10000         # 内存存储全局上限
      max-entries-per-trace: 500 # 单链路上限
    fallback-enabled: true   # logback 兜底开关
    pattern: "%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] [%X{traceId:-}] [%X{spanId:-}] %-5level %logger{36} - %msg%n"
    path: ./logs             # 兜底滚动文件目录
    max-history: 30          # 保留天数
    max-file-size: 100MB     # 单文件大小
```

### 4.3 扩展点（SPI）

| 扩展点 | 用途 | 接入方式 |
|---|---|---|
| `TraceIdGenerator` | 替换 traceId 生成算法（如雪花） | 注册同类型 Bean，默认 UUID 实现自动让位 |
| `TraceIdResolver` | 接入宿主自有 traceId 体系（优先级最高） | 注册 Bean，从请求中解析返回非空即采用 |
| `LogStore` | 日志持久化存储（如 Elasticsearch） | 注册 Bean，内存实现自动让位 |
| `WebSocketApiContributor` / `RpcApiContributor` | 自定义协议契约接入 `/apis` | 注册 Bean，控制器自动聚合 |
| `MdcWrappers` | commonPool/裸线程池的手动 MDC 包装 | `MdcWrappers.wrap(runnable/callable/supplier/...)` |

### 4.4 典型使用场景

1. **接入公司已有 traceId**：宿主已有 Filter 往 MDC 写 traceId → 配 `archimedes.trace.use-project-trace-id: true`；或注册 `TraceIdResolver` Bean 从自有请求头/上下文解析。
2. **排查一次请求的全链路日志**：从响应头（或 UI 调试面板）拿到 traceId → UI Trace Logs Tab 查询 → 请求线程 + `@Async` 线程 + 自定义线程池日志按时间线合并展示。
3. **REST 在线调试**：UI REST Tab 点击条目 "Try" → 参数已按契约预填 → Send → 看响应与耗时 → 点 "View trace logs" 直接看该次调用产生的服务端日志。
4. **生产接入 ES**：实现 `LogStore` 两个方法注册为 Bean 即完成替换，端点与 UI 无需任何改动。
5. **WebFlux 宿主**：契约展示（REST + RPC）与 UI 同等可用；链路追踪当前仅 Servlet 栈（响应式需 Reactor Context 传播，属后续规划）。

## 六、能力边界与后续规划

### 已明示的边界

| 边界 | 说明 |
|---|---|
| WebFlux 下的链路追踪/日志查询 | 响应式请求跨事件循环线程，MDC 快照模型不适用，需 Reactor Context 传播方案——契约展示已支持，追踪属后续版本 |
| `@ServerEndpoint` 须为 Spring Bean | 容器 SCI 直接注册（非 Bean）的端点不在扫描覆盖内 |
| tRPC 注解 FQCN | 按开源 tRPC-Java 假设；不匹配的内部版本自动不装配（安全退化，零报错） |
| 内存日志存储 | 重启即失、有界淘汰，定位开发/排障；生产持久化走 `LogStore` SPI |
| 跨线程自动覆盖范围 | Spring 容器管理的线程池；commonPool/裸线程池需 `MdcWrappers` 一行手动包装（无 javaagent 的物理边界，启动期有 WARN 预警） |

### 后续规划（Slice 14：发布与生态）

- 发布 Maven 中央仓库（待 Sonatype 账号与 GPG 密钥；发布后 Maven/Gradle 均可直接引入）
- Gradle 消费方验证与文档、Spring Cloud（网关/注册中心场景）兼容性验证
- 契约端点权限控制（避免生产环境敏感信息泄露）
